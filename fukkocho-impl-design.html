復興庁 実装設計書 v1.0

CTO(Fable)設計 → 統括レビュー ／ 実装＝Opus ／ 本番接触は全緑＋GOまで無し
統括メモ：Fableの設計。過去のファラBOT移行事故（古い土台で会話機能落として炎上）を踏まえ、最小差分・T0回帰テスト・本番実物との突合ゲート・テストch全緑まで本番切替禁止を全工程の原則に。やまちゃその「機能一覧5・6以外は反応させない」は、react.jsの復興庁chゲート（該当chなら即専用ハンドラへreturn）で、他の既存分岐に構造的に到達しない＝コード位置で自動的に実現（追加の除外ワードリスト不要）。末尾にやまちゃそ確認事項6件あり。

設計原則（全工程に優先）

  • P1：土台＝本番 /home/user/pharabot-staging の実稼働コード。丸ごと作り直し禁止・最小差分のみ。
  • P2：ローカル写し（worktree）は構造把握用。着手前に本番実物と必ず突合（写しはFirestore呼びが残る＝本番SQLite化後と乖離確実。差分を全部リスト化してから書く）。
  • P3：テストchで全挙動E2E全緑まで本番ch切替禁止。切替はch ID定数1箇所の変更のみで行える設計に。

1全体アーキ

[pharabot.service / node index.js]  ←既存プロセスに同居
 ├ botEntry.js（既存・無変更）
 ├ react.js messageCreate（既存＋復興庁分岐を最小挿入）
 │    └ 復興庁ch判定 → fukkocho/persona で生成
 ├ fukkocho/（新規・3バックエンド）
 │    ├ safety.js       … キル/レート/ログ/NG（3体共通の"関所"。安全装置だけ共有OK、台本・意図は非共有）
 │    ├ backend_phara.js … ファラBOT発起タイマー（本体clientで投稿）
 │    ├ backend_udatsu.js… 宇立部長（webhook送出）
 │    └ backend_famada.js… ファまだ（webhook送出）
 └ SQLite bot.db（既存＋テーブル2枚追加）
  • 3バックエンドは各自独立のタイマー（ジッター付き）。各tick＝「復興庁ch直近N件をREST取得→自分だけの判断ロジック＆プロンプトで投稿する/しないを決定→safety通過なら送出」。生成物を相互参照しない。連動はDiscord公開投稿のみ。
  • 宇立/ファまだはWebhook URL宛POSTのみ（受信なし＝一方通行が仕様通りの壁）。
  • 別プロセス化はしない（systemd/デプロイ/監視を増やさない最小差分）。プロセス内モジュール分離で「状態非共有」要件は満たす。将来bot化を見据え各backendはclient直依存でなく「読取fetch＋送出関数」を注入する形。

2既存コードへの最小差分（他ch挙動は不変）

新規 fukkocho/config.js

FUKKOCHO_CHANNEL_ID（.envから。テスト時=テストch／本番切替=env書換+restartのみ）、レート閾値、webhook名。

react.js への差分（3点だけ）

  • ①復興庁ゲート：messageCreate冒頭（保存の後）に if(channel==FUKKOCHO_CHANNEL_ID){ return handleFukkochoMessage(...) }。→ 以降の おファラオ/講評/教えて/ファ検知/画像ネタ 等 既存分岐に復興庁chは到達しない＝「①②③④・講評を鳴らさない」を分岐の物理位置で保証。保存(⑥)とロックはゲート前なので生きる。
  • ②handleFukkochoMessage（新規 fukkocho/handler.js）：
    ・リプ先＝ファラBOT本人 → fetchReplyChain → generateFukkochoReply（復興庁ペルソナ版）→ 返信。
    ・リプ先＝webhook投稿（宇立/ファまだ） → ファラBOTが人間表現で代弁（「まだお話しできないみたいで…！」。BOT/webhook/仕組み等はメタNGワード）。
    ・本文に「ファラBOT」→ 復興庁ペルソナ版で返信。
    ・いずれにも該当しない投稿は完全スルー（保存のみ）＝観測フィルタ。SHOWGO等 全サーバー共通ネタもゲートで到達しないので自動的に発火しない。
    ・応答は必ず safety.guard('phara_reply') 経由（会話枠 1h5件）。
  • ③messageReactionAdd（⑤）＝無変更。復興庁chでもそのまま動く。

プロンプト／起動

  • 既存プロンプト関数は触らず、復興庁用3種（reply/呼びかけ/代弁）を fukkocho/persona.js に新設、既存Geminiユーティリティだけ再利用。核＝既存口調維持＋「過疎に危機感を持ってなんとなく始めた素朴な発起人。応答はするが誘導・目標提示・メタ発言をしない」。
  • index.js ready後に require('./fukkocho').start(client)。FUKKOCHO_ENABLED=1 のときだけ起動＝本番で無効のままデプロイ可＝安全な段階リリース。

3宇立/ファまだ webhookバックエンド

  • 共通骨格（コードは共有OKだがインスタンス・状態・プロンプトは別）。タイマー：基準間隔＋ランダムジッター（叩き台：宇立 3〜8h／ファまだ 2〜6h で1回チラ見）。
  • tick：復興庁ch直近20件をREST取得（bot tokenで読むだけ。webhookは読めないので本体tokenのREST読取を注入）→ 各自の判断プロンプトで「今投稿する価値があるか(yes/no＋本文)」を1回のGeminiで判定生成 → noなら黙る（黙るのがデフォルト） → yesなら safety通過後 webhook POST。
  • 一方通行の演出：リプ機能を実装しない。自分宛リプが来ても本人は反応しない（代弁はファラBOT側）。
  • 宇立データ：reference/udatsu_dame_joushi_dataset.json からseverity低・非ハラスメント項目のみを実装時にホワイトリスト抽出して固定（実行時に全体を読ませない・抽出結果はやまちゃそレビュー必須）。ファまだ：浜田トーンの短いツッコミ/司会・容姿/能力イジり禁止をプロンプト＋NG両方に。
  • .env追加：FUKKOCHO_WEBHOOK_UDATSU/FAMADA, FUKKOCHO_CHANNEL_ID, FUKKOCHO_ENABLED（リポにコミットしない）。

4ファラBOT発起バックエンド

  • タイマー：叩き台 4〜10hジッター。
  • 入力＝直近のDiscord公開シグナルのみ（直近48hの投稿数・自分の直近投稿への反応有無・リアクション数）。長期記憶・目標・KPIを持たせない（状態ファイル無し・毎tickでch読み直すステートレス＝ゴールレスの実装的保証）。
  • 判断プロンプト：「過疎を寂しく思ってなんとなく見守りを始めた発起人。直近を見て言いたいことが自然に湧いたら短く1つ。無ければ黙る。目標・企画・呼びかけ連発・誘導はしない」。送出は safety.guard('phara_initiate')（各自1日3枠）。

53層安全装置（fukkocho/safety.js）

  • ①キルスイッチ：/var/lib/pharabot/HALT_FUKKOCHO を全送出直前に同期チェック→あれば黙ってログのみ。権限次第で ./HALT_FUKKOCHO にフォールバック（どちらを正にしたか実装時明記）。
  • ②サーキットブレーカー：SQLite送信ログをCOUNTで判定（メモリカウンタにしない＝再起動で消えない）。閾値(初期・env上書き可)＝発起系合算 1h3/24h8・各persona 24h3／会話応答 別枠1h5／類似連投ガード(直近5件と先頭50字一致で棄却)。超過＝該当バック自己停止＋DM通知1通(クールダウン付)。
  • ③ログ＆NGチェック：全送出を fukkocho_posts に記録。送信前に本文をNG判定（特定住民名言及/イジり／メタ語彙「BOT/AI/webhook/プログラム/仕組み」等の正規表現＋必要ならGemini軽量判定）→NGなら棄却+reason記録。既存is_bot=1ログと整合（personaは別テーブルで持ち既存スキーマは触らない）。

7テスト計画（全緑まで本番切替禁止）

環境＝FUKKOCHO_CHANNEL_ID=テストch、テスト用webhook2本を別途作成（本番webhookはこの段階で使わない）。

  • T0 回帰（最重要）：復興庁ON状態で他chの既存挙動が不変を実メッセージ確認（おファラオ+PP／リプ会話／「ファラBOT」反応／画像ネタ／リアクション⑤／保存⑥）。1つ欠けたら差分を疑う。
  • T1 会話：BOT投稿にリプ→復興庁ペルソナで返る（誘導・メタ無し）。「ファラBOT」含む→返る。PP/保存も付く。
  • T2 代弁：宇立webhook投稿に人間リプ→ファラBOT人間表現で代弁。メタ語ゼロ。
  • T3 フィルタ誤爆なし：テストchで「SHOWGO」「無駄」等→一切反応しない（保存のみ）。
  • T4 発起/一方通行：タイマー短縮で24h空回し→3体が各自投稿・掛け合いリプしない・沈黙判断が効く（毎tick投稿にならない）。
  • T5 安全装置：HALT作成→全停止／レート閾値1で超過→自己停止+DM／NGワード故意生成→blocked_ng記録。
  • 本番切替チェックリスト：T0〜T5全緑／本番と写しの突合済／本番webhook2本作成・env設定／タイマー・レートを本番値に／HALT動作を本番でも1回確認／切替=env書換+restart／切替後24h監視・異常即HALT。

⚠ やまちゃそ確認事項（6件）

  • 1. 写しの鮮度：ローカル写しはFirestore呼びが残る＝本番SQLite化後と乖離確実。M1のdiffレポートを承認ゲートにする（＝実装最初に差分を出して見せる）。
  • 2. タイマー初期値（宇立3-8h／ファまだ2-6h／ファラ4-10h）はFableの叩き台。頻度感の好みは？
  • 3. 宇立データセットの抽出結果（低severity・非ハラのみ）は投稿前にお前の目視レビュー必須（ハラ混入防止）。
  • 4. 復興庁chで「おファラオ」PP付与を殺してよいか：現設計はゲートで既存ネタ全停止＝おファラオ挨拶の+10PPボーナスもこのchでは付かない（投稿の基本保存・PP＝⑥は付く）。①扱いで切る、で合ってる？付けたければゲート前に例外1行。
  • 5. サーキットブレーカー発報のDM宛先＝お前のDMでいい？
  • 6. Gemini呼び出し増によるAPIコスト微増（3体×チラ見。判断+生成を1呼びに纏めて抑制済）。量産前に単価確認推奨。
