ファラBOT Firestore→SQLite移行 詳細設計書（案A：既存土台・その場差し替え）

方針一言：本番で動いているPharaBOT2はそのまま。触るのは「保存先」と「PP処理の呼び出し方」だけ。会話(Gemini返信)は1行も変えない。

1. 全体方針

  • 土台＝現本番コード(PharaBOT2)。新フォルダは作らない。
  • 前回事故（別フォルダに書き直し→comment.js未移植でBOT沈黙）の再発防止として、「新規ファイル作成は下記3つのみ・既存ファイルの書き換えは2ファイルのみ」を上限とする。

改修対象（これ以外は無改造）:

  • 書き換える既存ファイル（2つ）
  • functions/firestoreService.js → 中身をSQLite実装に置換（関数名・引数は不変）
  • Discord/botEntry.js → 起動時にDB open＋cron登録を数行追加
  • 新規に置くファイル（3つ・全部pharabot-gceからのコピー流用）
  • db/ (better-sqlite3接続層 = pharabot-gceのdb.js + migrations)
  • domain/ppService.js (PP計算 = pharabot-gceのppService.js + pp.js)
  • jobs/scheduled.js (定期5本のcron化。中身はrevel.jsのロジック移植)
  • comment.js・react.jsは呼び出し先が同名関数のままなので原則ノータッチ（PP発火の1行追加のみ）。
  • 掃除・ファイル分割・スキーマ刷新はやらない（安定稼働後に別案件）。

2. DB差し替え（firestoreService.jsの中身入れ替え）

ファイル名・関数名（＝呼び出し方の形）を完全に維持して中身だけSQLiteにする。呼び出し側(react.js/comment.js)は無改修で動く。

  • saveMessageToFirestore(message) → messagesテーブルへINSERT（同じ項目）。保存直後にppService.onMessageSaved()を同期で呼ぶ。
  • incrementReactionCount(messageId) → total_reactions を+1。
  • addGreetingPP(...) → 今はHTTPでCloud Functionsを呼んでるが、同一プロセス内のppService直呼びに変更（ネット越し→関数呼び出し。速く確実に）。
  • recordGreetingEvent / fetchRecentGreetingEvents → greeting_eventsテーブルのINSERT/SELECT。二重保存を1テーブルに統合。
  • claimScheduleExecution(key) → schedule_locksに INSERT OR IGNORE（二重実行防止）。
  • getDb() → SQLiteハンドルを返す。

部品流用：pharabot-gceの db.js（WALモード・migration自動適用は実装済み）と初期スキーマSQLをコピー。無い表(schedule_locks)は1枚追加。

DBファイル：GCE上 /var/lib/pharabot/bot.db（環境変数で変更可、こもやんと同方式）。

3. PP処理のBot内移設

現状：Firestoreに保存→クラウド側が自動検知(onMessageWrite)→PP計算。

移行後：Botがメッセージ保存した直後に、自分でPP計算関数を呼ぶ（自動検知の仕組みはSQLiteに無いため。呼び忘れリスクは保存関数の中に組み込んで排除）。

  • 接続点：saveMessageToFirestore の末尾で ppService.onMessageSaved() を呼ぶ1箇所のみ。comment.js/react.jsは触らない。
  • ppServiceに残す機能（revel.jsのロジックを移植、pharabot-gce実装分を土台にrevel.jsと等価確認）:
  • 日次キャップ（1日に稼げるPP上限）
  • レベル計算・レベルアップ時のWebhook通知（お祝い投稿）→通知URL等のenv引き継ぎ
  • 100投稿ごとのDNA解析（OpenAI呼出）→カウンタ到達で発火。失敗してもPP付与は成功扱い。

4. 定期処理（onSchedule 5本の置き換え）

revel.jsの5本（週次レポ 月9:00 / 月次レポ 1日9:00 / 週次ボーナス 月6:00 / 月次ボーナス 1日6:00 / 年次ボーナス 2/1 6:00、全てJST）を、Bot内node-cron（プログラム内の目覚まし時計。PharaBOT2は既に利用実績あり）で再現。

  • jobs/scheduled.js に5本を移植、botEntry起動時にcron登録。
  • 各実行冒頭でロック確認（再起動直後の二重実行防止）。
  • systemd timerは使わない（Bot本体が常駐してるので同居が簡単）。

5. 会話(comment.js)/Gemini — 無改造

  • comment.js（約1500行）は1行も変更しない。これが案Aの核。
  • 引き継ぐenv：GEMINI_API_KEY、OpenAIキー、Webhook URL、Discordトークン → GCEの .env ＋常駐設定に転記。切替前にenv一覧の棚卸しチェックリストを作る（キー漏れ＝沈黙事故の典型）。
  • comment.jsがfirestoreServiceの関数を呼ぶ箇所はシグネチャ不変でそのまま動く。comment.js/react.js内に直接firebase参照が無いかをgrepで最終確認（受け入れ前の必須チェック）。

6. データ移行・カットオーバー（切替）手順

  • 移行スクリプト＝pharabot-gceの migrate_firestore.js（--sinceバグ修正済み）を流用。messages / PPログ / users / greeting履歴をSQLiteへ。
  • 手順（本番は最後の数分だけ止める）:
  • 事前：GCEに全部構築→過去データ全量移行→ステージング（テスト用ch）で受け入れ条件を確認
  • 深夜 2:00：旧Bot停止 → --sinceで差分だけ再移行（数分）→ 新Bot起動 → 疎通確認（おファラオ1発）
  • 旧側は消さない：Cloud Run/Functionsは停止のみで温存。異常時は旧Bot再起動で5分でロールバック（Firestoreデータは無傷）
  • 1週間安定を確認してから旧環境を畳む

7. 受け入れ条件（ステージングで全部緑になるまで本番切替しない）

  • 「おファラオ」にGemini返信が返る（会話が生きている＝前回事故の再発チェック）
  • メッセージ投稿でPPが付く（日次キャップ・挨拶PPも）
  • 週次ボーナスがcron発火で動く（時刻を仮変更して強制発火テスト）
  • 新旧PP一致：移行後SQLite集計とFirestore集計を全ユーザー突き合わせ、差分ゼロ
  • Bot再起動後もデータが残り、定期処理が二重実行されない

8. リスクと落とし穴

  • 最大リスク＝「動いてるコードの見落とし」：revel.jsにはaddGreetingBonus/getLeaderboard等のHTTP関数もある。Botや外部から叩かれてる口を全部棚卸しし、移設or廃止を明示。
  • comment.js内の隠れfirestore直参照（想定は無いが、grep確認をマイルストーン条件に）。
  • DNA解析・レポートのOpenAIコスト/キー引き継ぎ漏れ。
  • SQLite単一ファイル破損 → こもやんと同じ日次バックアップを初日から。
  • e2-microのメモリ：ステージングで常駐メモリ実測。

9. マイルストーン（Opusコーダーに渡す粒度）

  • M1 DB層差し替え：db層移植＋firestoreService.js中身置換＋PP発火接続。完了条件＝受け入れ①②相当がローカルで緑＋firebase参照grepゼロ
  • M2 定期処理移植：jobs/scheduled.js（5本＋ロック）＋getLeaderboard等HTTP口の棚卸し対応。完了条件＝強制発火テスト緑
  • M3 データ移行＆検証：migrate実行＋新旧PP突き合わせ。完了条件＝差分ゼロレポート
  • M4 GCE構築＆ステージング：e2-micro＋常駐＋env転記＋バックアップ。受け入れ①〜⑤全緑
  • M5 カットオーバー：深夜切替＋1週間監視＋旧環境温存

施主への確認ポイント（2つだけ）

  • 切替深夜の日程（いつやる？）
  • getLeaderboard等のHTTP機能を現在使っているか（使ってなければ移設不要でスコープが軽くなる）
